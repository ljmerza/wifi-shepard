## Executive summary

The codebase is in good health, and the audit's own numbers say so: of 41 reported findings, **zero were critical** and only **three were high**. There is no structural rot here — no injection, no auth bypass, no data-loss path, no N+1. The test suite (~7,240 LOC against ~3,970 source LOC) is genuinely well built, driving real HTTP through `aioresponses` and real SQLite through `TestClient` rather than mocking behavior away, and the nine ADRs document trade-offs so thoroughly that five separate findings were refuted *because the ADRs had already reasoned through them*.

The three high findings share one theme: **safety controls that fail silently and open.** The allowlist — the daemon's primary protection against kicking a device you told it not to touch — is an exact-match string test, so an operator who types a MAC in uppercase (the way it is printed on the device label) gets an entry that silently never matches. Config validation errors repr the raw YAML *after* `${VAR}` interpolation, so a one-character indentation slip writes the live UniFi admin password into `docker logs`. And SIGHUP — the documented mechanism for applying config changes, required by four separate ADR acceptance criteria — never reaches the daemon at all, because `uv` sits at PID 1 and does not forward the signal. Each of the three fails in the direction of *doing the wrong thing quietly*, which is precisely the failure mode a re-roaming daemon can least afford.

The dominant secondary theme is **inconsistency rather than absence**: the repo already knows how to do the right thing and does it in one place but not another. `normalize_mac` exists and is used for an advisory log warning, but not for the allowlist gate that decides whether to deauth a camera. `Dockerfile.ui` creates a `nologin` user and drops privileges with a test enforcing it; the daemon `Dockerfile` runs as root with no such test. The UI is opened `mode=ro` with `PRAGMA query_only` as defense-in-depth, while the compose file mounts the config file's own directory read-write next to its `:ro` mount. Most fixes here are small and consist of applying an existing pattern one layer over.

Nothing found contradicts the project's v1 posture. The most consequential caveat is that `dry_run` is the gate standing between these findings and real-world effect — see Methodology for what that means and what this audit did not cover.

## Confirmed findings

### 1. [high] SIGHUP config reload never reaches the daemon: uv 0.5 as PID 1 does not forward SIGHUP

- **Where**: `Dockerfile:14` (deployment, security)
- **Issue**: CMD ["uv", "run", "python", "-m", "wifi_shepard"] makes uv PID 1 with Python as a child (verified: uv spawns, does not exec). Empirically tested the pinned line's terminal release, uv 0.5.31: SIGTERM IS forwarded to the child with exit-code propagation, but SIGHUP is NOT — uv has no SIGHUP handler (outside a container it dies of the default disposition, exit 129, and the child never sees the signal). As container PID 1, a handler-less signal is discarded by the kernel, so `docker kill -s HUP wifi-shepard` is a silent no-op.
- **Failure scenario**: Operator edits config.yaml (e.g. adds a MAC to allowlist: or flips dry_run) and sends SIGHUP per the documented reload path -> uv (PID 1) discards the signal, _on_sighup never fires, daemon keeps running with the stale config and no log line indicates the reload was missed -> device that should now be protected keeps getting kicked.
- **Suggested fix**: Make Python PID 1: CMD ["/app/.venv/bin/python", "-m", "wifi_shepard"] (uv sync --frozen already materializes /app/.venv at build time, so uv run adds nothing at runtime). Alternatively add `init: true` in compose AND bump uv to a version verified to forward SIGHUP (0.11.26 forwards both SIGHUP and SIGTERM — verified).
- **Verifier's evidence**: CONFIRMED by independent empirical reproduction, not just code reading. (1) Citation correct: Dockerfile:14 is CMD ["uv","run","python","-m","wifi_shepard"] behind the floating ghcr.io/astral-sh/uv:0.5 pin (Dockerfile:5), and docker-compose.yml has no init: true, so uv is container PID 1. (2) Downloaded the actual uv 0.5.31 release binary (terminal 0.5.x release) and tested: `uv run` spawns python as a child (verified via process tree, PID 2701555 -> 2701562), does not exec.

### 2. [high] Config validation errors echo interpolated secrets (UNIFI_PASSWORD / HA_TOKEN) into container logs

- **Where**: `src/wifi_shepard/config.py:115` (security)
- **Issue**: Env interpolation runs at load_config_from_path (config.py:729) BEFORE structural validation, and several validators embed the full raw value in their ValueError message: _require_sequence (line 115, '{value!r}'), _require_mapping_items (line 124, '{item!r}'), _build_home_assistant (line 335, '{raw!r}'), _build_reboot (line 393), _build_reboot_cooldown/proactive/reactive (456/472/491), and _build_quiet_hours (line 547). A common YAML slip — writing 'controllers:' as a mapping instead of a list (dropping the '- ' dash) — makes _require_sequence raise 'controllers must be a YAML list, got dict: {...password: <live UNIFI_PASSWORD>...}'.
- **Failure scenario**: Operator sets UNIFI_PASSWORD in env/wifi-shepard.env, then mis-indents config.yaml so 'controllers:' holds a mapping (a one-character YAML mistake). On the next start or SIGHUP, ValueError('controllers must be a YAML list, got dict: {..., \'password\': \'<actual admin password>\', ...}') is written verbatim to docker logs; the log line then flows to any log shipper, and is precisely the line an operator copy-pastes into a GitHub issue when asking why startup failed.
- **Suggested fix**: Never include raw values in structural error messages once interpolation has run: report only the offending key path and the Python type (e.g. 'controllers must be a YAML list, got dict'), or redact/truncate values, or run structural shape checks BEFORE _walk_and_interpolate and only interpolate after the shape is known-good.
- **Verifier's evidence**: CONFIRMED by independent code reading and a live reproduction. (1) Order of operations: config.py:729 runs `_walk_and_interpolate(data)` (env-secret substitution) before any structural shape check; `controllers` is only shape-checked afterward at config.py:754-758 via `_require_sequence`. (2) The cited error paths do embed raw values: config.py:115 (`{value!r}` in _require_sequence), 124 (`{item!r}`), 335 (`{raw!r}` in _build_home_assistant), plus 393/456/473/491/547 — all verified by grep.

### 3. [high] Allowlist check is case/format-sensitive and fails open — allowlisted MACs can still be kicked

- **Where**: `src/wifi_shepard/scorer.py:64` (correctness)
- **Issue**: scorer.ingest() gates the entire kick path with `if mac in self.config.allowlist:` — a raw, exact-match membership test against the unnormalized tuple stored at config.py:717 (`allowlist=tuple(allowlist)`). This is the ONLY allowlist enforcement point in the kick pipeline (grep confirms: actor.py never re-checks it).
- **Failure scenario**: Operator adds `B8:AB:62:AF:D4:29` (uppercase, as printed on the device label / router UI) to allowlist:, or a UniFi firmware update starts reporting uppercase MACs while the config holds lowercase. `'b8:ab:62:af:d4:29' in ('B8:AB:62:AF:D4:29', ...)` is False, so the supposedly-protected attic camera is scored, flagged bad-state, and deauth-kicked — silently, with no warning that the allowlist entry never matched anything.
- **Suggested fix**: Normalize once at parse time: in config.py build_config, store `allowlist=tuple(normalize_mac(m) for m in allowlist)` (reusing reboot.normalize_mac), and normalize the client MAC in scorer.ingest before the membership test (`if normalize_mac(mac) in self.config.allowlist`). Optionally validate MAC format at load (fail-closed posture).
- **Verifier's evidence**: Independently confirmed every load-bearing claim by reading the code. (1) src/wifi_shepard/scorer.py:64 does a raw exact-match `if mac in self.config.allowlist:` with no normalization of either side. (2) config.py:717 stores `allowlist=tuple(allowlist)` unnormalized, and config.py:750 (`_require_sequence`) does no MAC format/case validation.

### 4. [medium] Daemon container runs as root; UI hardening (non-root user, test enforcement) was never applied to the daemon

- **Where**: `Dockerfile:14` (deployment, security)
- **Issue**: The daemon Dockerfile has no USER directive, so uv and the Python process run as uid 0. Dockerfile.ui:12 creates a nologin uid-1000 user and drops to it at line 20, with its rationale comment applying equally to the daemon ('root brings nothing useful'), and tests/ui/test_ac_1_compose_and_dockerfile.py:50-62 enforces non-root for the UI only. The daemon needs only outbound HTTPS to the UniFi controller / Home Assistant and write access to /data — nothing requiring root. The compose fragment adds no compensating `user:`, `read_only:`, or `cap_drop:` on either service.
- **Failure scenario**: Any code-execution bug reachable through parsing hostile UniFi controller responses (aiounifi==85 is exact-pinned and will never take security patches automatically) executes as root in the container: full default capability set, root-owned writes into the bind-mounted ./volumes/wifi-shepard on the host, and a materially larger kernel attack surface for container escape than the uid-1000 UI sidecar.
- **Suggested fix**: Mirror Dockerfile.ui: useradd a uid-1000 nologin user, chown /app, add USER before CMD (ensure /data is writable by that uid — chown the host dir once or document PUID). Add `cap_drop: [ALL]`, `read_only: true` (with /data left writable) and `security_opt: [no-new-privileges:true]` to both services in the fragment, and extend the non-root Dockerfile test to cover the daemon Dockerfile.
- **Verifier's evidence**: Independently confirmed. Dockerfile (daemon) has no USER directive — CMD at line 14 runs as uid 0 on python:3.12-slim. Dockerfile.ui:12 creates a uid-1000 nologin user and drops to it at line 20, with a rationale comment (lines 9-11) that applies equally to the daemon. tests/ui/test_ac_1_compose_and_dockerfile.py:50-62 enforces non-root for Dockerfile.ui only; no test covers the daemon Dockerfile.

### 6. [medium] Stale pending-BTM entry survives a recovered no-roam client and later fires an ungated, uncounted deauth

- **Where**: `src/wifi_shepard/actor.py:255` (correctness)
- **Issue**: check_post_kick_outcome clears the pending-BTM map only when `roamed and pending['mechanism'] == 'btm'` (line 255-256). The comment claims the no-roam case is consumed 'on this same cycle via the deauth_fallback path' — but handle() only runs when the scorer flags bad-state THAT cycle. If a BTM'd client stays on the same AP and its window is no longer bad (kick reset its stats / metrics improved), the `_btm` entry persists indefinitely.
- **Failure scenario**: kick_mechanism auto/btm; client X bad on AP A -> BTM sent, _btm[X] set. Next cycle X is on A but healthy (post-kick stats reset) -> kick_no_roam logged, _btm kept. Three days later X degrades on A again -> instead of a fresh gated kick, handle() immediately deauths X under the 3-day-old attempt_group, skipping cooldowns (shipped: 300..86400s), hourly/daily caps, and per-AP cap, and the kick never counts toward any future cap.
- **Suggested fix**: Expire the pending-BTM entry when it isn't consumed on the immediately following cycle: e.g. in check_post_kick_outcome, on the no-roam case mark the entry consumable-this-cycle (store a cycle/timestamp in set_btm and have handle() treat entries older than ~2x poll_interval as stale: clear them and take the fresh-kick path instead).
- **Verifier's evidence**: CONFIRMED by independent code reading. actor.py:255-256 clears the pending-BTM entry only on (roamed AND mechanism=='btm'); the no-roam case relies on handle() consuming it the same cycle (comment at 249-254), but scanner.py:80-83 only calls handle() when scorer.ingest returns a bad-state decision.

### 7. [medium] HA long-lived token sent as Bearer over cleartext http:// — permitted by validation, used by the shipped config, never warned about

- **Where**: `src/wifi_shepard/config.py:342` (security)
- **Issue**: _build_home_assistant (config.py:342-343) accepts http:// URLs, and the notifier (src/wifi_shepard/notify/home_assistant.py:60-63) then sends 'Authorization: Bearer <HA_TOKEN>' in cleartext on every kick/quarantine notification. Unlike verify_ssl: false — which carries an explicit inline comment documenting the trade-off and how to restore the secure default — no ADR, README, or config comment acknowledges cleartext token transport, and no startup warning is emitted.
- **Failure scenario**: This project's own premise is that the LAN hosts cheap, misbehaving off-brand IoT devices. The token is long-lived, so one capture is persistent access.
- **Suggested fix**: At minimum log a startup warning when home_assistant.url is http:// (mirroring the verify_ssl documentation pattern), and document the trade-off in config.example.yaml. Better: require https:// unless an explicit allow_insecure_http: true opt-out is set, mirroring the fail-closed posture used everywhere else in config.py.
- **Verifier's evidence**: CONFIRMED by direct code reading. (1) config.py:342-343 accepts http:// URLs with no warning or opt-out gate — the only warnings in the config load path (config.py:697/702/708) are reboot-eligibility related. (2) notify/home_assistant.py:63 sends 'Authorization: Bearer <HA_TOKEN>' on every notify POST, so an http:// URL means cleartext long-lived-token transport per kick/quarantine.

### 12. [medium] Scanners run sequentially in one for-loop; a slow controller starves the others, contradicting PLAN.md's concurrent-per-controller design

### 32. [low] UI auth middleware returns 500 (unhandled TypeError) instead of 401 for non-ASCII bearer tokens

- **Where**: `src/wifi_shepard_ui/app.py:185` (security)
- **Issue**: secrets.compare_digest raises TypeError when given str arguments containing non-ASCII characters. Starlette decodes header values as latin-1, so any Authorization header whose token part contains a byte >= 0x80 reaches app.py:185 as a non-ASCII str, and compare_digest(presented, expected_token) raises inside the middleware. FastAPI converts the unhandled exception into a 500 Internal Server Error and logs a full traceback. Auth is not bypassed (the exception aborts before call_next), but the auth layer violates its fail-closed 401 contract and gives unauthenticated clients a trivial 500/log-spam primitive.
- **Failure scenario**: With WIFI_SHEPARD_UI_TOKEN set, an unauthenticated client sends 'Authorization: Bearer \xa1x' (any non-ASCII byte) to any route. Instead of the intended 401 'unauthorized', the server raises TypeError('comparing strings with non-ASCII characters is not supported'), returns 500, and writes a traceback to the container logs on every such request.
- **Suggested fix**: Compare bytes instead of str: secrets.compare_digest(presented.encode('utf-8'), expected_token.encode('utf-8')) — encoding a latin-1-decoded header to UTF-8 never fails — or wrap the comparison in try/except TypeError and return the 401 response.
- **Verifier's evidence**: Confirmed by direct code reading and an end-to-end reproduction. src/wifi_shepard_ui/app.py:173-188: the bearer_token_auth middleware wraps no try/except around secrets.compare_digest at line 185, and the short-circuit order (scheme check, then `not presented`, then compare_digest) means any 'Bearer <non-ASCII>' header reaches compare_digest as a non-ASCII str.

## Methodology

**Shape.** Five Fable finder agents, one per dimension, each given an explicit file list and repo-specific hot spots drawn from a prior inventory pass rather than a bare "audit this repo" instruction. Every finding was then handed to an independent Fable skeptic agent prompted to *refute* it — instructed to default to refuted when uncertain, and to let a finding survive only on independent confirmation from reading the code. Finders and verifiers ran as a pipeline with no barrier, so each dimension's findings verified while other finders were still sweeping.

**Cost.** 46 agents, ~3.0M subagent tokens across both passes, ~712 tool calls, ~18 min wall clock. The cap of 15 findings per dimension was never reached (max was 11), so nothing was truncated.

**False-positive control.** Every finder prompt carried an enumerated list of the trade-offs the nine ADRs explicitly accept — in-memory rate-limit state lost on restart (ADR-0004), two `kick_events` rows per BTM+fallback kick (ADR-0003), `ap_cu_total == 0` conflating unknown with idle (ADR-0008), the deliberately absent concrete `Rebooter` (ADR-0006), the UI's read-only, auth-off-by-default posture (ADR-0002), and others — with instructions to report them only if the *implementation* of the trade-off is broken. This is why the report is not full of re-litigated design decisions.

**Refutation rate: 5 of 41 (12%).** This is low for an adversarial pass and should temper confidence somewhat: a rubber-stamping skeptic produces the same number as a rigorous one. Three things argue it was rigorous. The refutations are substantive rather than pro-forma — the `db.py:195` transaction-interleaving race was killed on the grounds that the concurrent writer *cannot exist in production* because the reboot scheduler never starts without a concrete `Rebooter`, which is exactly the ADR-0006-aware reasoning the prompt asked for. The skeptics also killed two leads carried in from the inventory pass (the `FakeHANotifier` signature drift and the `oui.py` malformed-MAC path). And the finders were handed pre-verified leads, so a high survival rate is partly by construction. Even so, 0/6 refutation in security and 0/8 in quality is the weakest evidence in this report.

**Manual spot-checks.** All three high findings were independently verified by reading the cited code directly, not taken on the agents' word: `scorer.py:64` (the bare `in` test, and `config.py:750` storing the allowlist unnormalized while `config.py:690` normalizes for a mere warning), `config.py:115` (`{value!r}`, with `_walk_and_interpolate` at line 729 confirmed to run before the shape check at 755), and `Dockerfile:14` (`CMD ["uv", "run", ...]`, no `USER`, no `HEALTHCHECK`, against `Dockerfile.ui:12,20` which does drop privileges). The compose mount and `expose: 8683` vs `EXPOSE 8080` mismatches were also confirmed by hand. The `Dockerfile:14` verifier went further than required and empirically tested a downloaded uv 0.5.31 binary, observing SIGHUP kill uv with exit 129 while the child process never received it.

**Known weaknesses in this audit.** (1) Four verifiers — `scorer.py:64`, `main.py:129`, `db.py:47`, `views.py:50` — ran while the safety classifier was unavailable; `scorer.py:64` was hand-checked in response, the other three are low/medium. (2) The post-processing merges findings within ±3 lines of each other in the same file, which collapsed three genuinely distinct defects into unrelated anchors; this was caught and reversed, and the affected entries (daemon-runs-as-root, the scan-loop resilience test illusion, the `:ro` mount subversion) are restored above as their own findings. A tighter window would not have needed the rescue. (3) Severity is the finder's, adjusted by the skeptic where it disagreed — it is a judgement, not a measurement.

**Not covered.** Findings are overwhelmingly from reading the code, with three exceptions where a verifier went further on its own initiative: the `Dockerfile:14` skeptic downloaded and signal-tested a uv 0.5.31 binary, the `config.py:115` skeptic reproduced the secret leak end-to-end against a live env var, and the `test_daemon_multi_controller.py` skeptic ran that test under log capture to observe the swallowed `AttributeError`. Otherwise: runtime behavior against a live UniFi controller, actual `aiounifi` upstream behavior, performance under real client load, and the UI rendered in a browser were all out of scope. Dependency CVEs were not scanned — finding 23 is that nothing scans them, not a scan result. The audit read the working tree, including the uncommitted changes to `src/wifi_shepard/controllers/unifi.py` and `tests/test_unifi_controller.py`, which are the subject of finding 9.

**Standing caveat.** Several findings are gated in practice by `scanner.dry_run`. That gate is the reason the allowlist bypass has probably not bitten anyone yet — it is not a reason the bypass is not real. It fails open the moment actions are enabled.
